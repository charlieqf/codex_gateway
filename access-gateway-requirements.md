## 1. 问题陈述

订阅持有者拥有某项**付费 AI 服务订阅**——典型如 ChatGPT / Codex、Claude Pro / Max、Kimi 订阅、DeepSeek 订阅等，也可能是同时持有多项。希望多台设备、乃至少数受信用户能够通过持有者的订阅使用 AI 能力，**但不要求每台设备重新登录订阅账号**。同时：

- 持有者需要对"谁能用、用多少、用多久、能做什么"保持控制
- 订阅原始凭据不下发到任何终端
- 失控时能快速止损

**本文档刻意对底层 AI 提供商保持中立**。本系统不绑定任何单一厂商；但任何 AI 服务要接入，都必须先验证其是否支持"服务端保持登录态、用已登录身份代表用户调模型"的使用方式（无论是 OAuth token、subscription bearer、还是 API key），并能满足续期、流式响应、错误归一化和会话延续等 adapter 合同。MVP 固定覆盖一个已验证提供商的典型路径，v2+ 再考虑多提供商并存（见 §10 决策 D9）。

本需求文档描述为满足上述诉求所需的系统功能与行为，不涉及具体实现方案，也不偏好任何特定提供商。

## 2. 范围与非范围

**在范围内**：

- 终端通过访问凭据使用订阅能力
- 持有者对访问凭据的生命周期管理（签发、例行轮换、吊销、限流、分权）
- 多设备、多用户场景覆盖；多订阅池化作为 v2 能力
- **对底层 AI 提供商中立**：访问凭据体系与管理语义保持中立；MVP 仅覆盖一个已验证提供商，v2+ 支持多提供商 adapter 并存
- 同一用户跨设备会话延续
- 服务端对原始订阅凭据的托管
- 客户端访问体验
- 持有者视角的使用情况观察与审计（元数据层，非内容）

**不在范围内**：

- 匿名/公开访问、多租户 SaaS、对外付费运营
- 跨订阅的会话数据迁移
- 医疗合规审计（HIPAA / 等保 / 患者数据追溯）
- 第三方 AI 服务自身条款的法律判断（由持有者自行评估）
- 为不支持"服务端代理登录态"的 AI 服务发明绕过方案

**技术边界收紧**：

- MVP = 单提供商、单订阅、多访问凭据；多订阅池化和多提供商并存不进入 MVP 验收。
- 提供商中立 = 网关核心不绑定具体提供商 + 通过 adapter 接入；不等于所有提供商零成本接入。
- 会话归属以"逻辑用户 / subject"为边界：默认一把新凭据对应一个 subject；例行轮换时新旧凭据可显式归属同一 subject。
- 不相关凭据之间会话历史不跨越；同一 subject 下的轮换凭据共享会话历史。
- "订阅凭据零下发"按可测口径验收：协议、配置、日志、持久化目录和抓包中不得出现原始订阅凭据；不承诺证明任意终端内存无痕。

## 3. 角色

| 角色 | 说明 |
|------|------|
| **订阅持有者** | 拥有付费 AI 订阅、运维访问网关、签发/吊销访问凭据的人 |
| **终端用户** | 持有访问凭据、通过客户端使用 AI 能力的人。可能是持有者本人的其他设备用户，也可能是受信同事/家人/学生 |
| **系统运营者** | 负责访问网关本身的部署、监控、事故响应的人。通常与订阅持有者是同一人 |

## 4. 使用场景

每个场景按以下格式描述：**情境** → **触发** → **期望体验** → **边界/变体**。场景分 MVP（S1–S7）、v2（S8–S9）、v2+（S10）三档，再加跨档次共性场景（S11–S13）。

### MVP 场景

#### S1：订阅持有者跨自己设备零重登

| 维度 | 内容 |
|------|------|
| 情境 | 持有者在自己家里、公司、笔记本、平板多处工作，每处都有安装客户端 |
| 触发 | 切换设备打开客户端 |
| 期望 | 无需再登录订阅服务，直接继续使用 AI 能力 |
| 边界 | 长期设备（家 / 公司）用长期访问凭据；一次性设备（演讲用借来的机器）参见 S2 |

**典型画像**：
- 医生上班用办公台式、下班用家里笔记本，查阅同一份医学证据
- 研究者在实验室、办公室、家里查文献，希望同一套历史跟随

#### S2：临时终端快速接入

| 维度 | 内容 |
|------|------|
| 情境 | 临时借用一台陌生机器（图书馆、酒店商务中心、会议室共享电脑、同事工位） |
| 触发 | 持有者或被授权者当场安装客户端、输入短期访问凭据 |
| 期望 | 立即可用；使用结束后凭据自动失效；对该机器无长期副作用 |
| 边界 | 凭据建议设短过期（小时级）；离开前即便忘记主动清理，凭据到期自动失能 |

**典型画像**：
- 学术会议间隙在会场电脑查一条医学问题
- 出差住酒店忘带笔记本，用商务中心电脑应急工作

#### S3：受信小圈子共享

| 维度 | 内容 |
|------|------|
| 情境 | 持有者愿把订阅能力分享给少数受信个人（家人、学生、小团队） |
| 触发 | 持有者为每位受信者签发独立访问凭据，带个性化标签和限流 |
| 期望 | 每人独立身份、互不干扰；观察谁在用、用得多、最后一次什么时候用；任一位有问题可独立处理而不牵连其他 |
| 边界 | 合规责任由持有者自行判断（参见 §7.1） |

**典型画像**：
- 主治医师给几个研究生一人一把凭据查证据
- 医学教学班十几个学生一人一把凭据做问诊练习
- 多地协作的研究小组，每位成员独立凭据

#### S4：按用途分级授权

| 维度 | 内容 |
|------|------|
| 情境 | 同一订阅用于不同风险等级的用途：医学问诊（低风险工具集即可）vs 代码实验（需要 shell/文件编辑）|
| 触发 | 持有者按用途签发不同"scope"的访问凭据 |
| 期望 | 低 scope 凭据无法触发高风险操作；权限差异由服务端强制，客户端无法绕过 |
| 边界 | MVP 至少覆盖 medical（仅问答类工具）和 code（全工具）两档；更细的角色分级为 v2 |

**典型画像**：
- 持有者给自己签发 code 凭据做开发实验；给同事 / 学生签发 medical 凭据做临床问答
- 持有者担心某凭据用户误用 shell 命令，只发 medical

#### S5：演示 / 临时授权

| 维度 | 内容 |
|------|------|
| 情境 | 演讲、课堂、工作坊需要观众短时间亲自体验 |
| 触发 | 持有者批量签发短期（小时级）、低限流（每分钟少量请求）凭据，分发给听众 |
| 期望 | 听众立即体验；过期 / 超频自动失效；持有者无需手工回收；单个听众用完配额不影响其他听众 |
| 边界 | 凭据过期后相关会话数据归档；持有者可随时提前批量吊销 |

**典型画像**：
- 学术演讲中演示"医学证据 AI"，现场发 20 把凭据让观众试用
- 教学课堂课时发凭据，下课自动失效

#### S6：凭据失控应急

| 维度 | 内容 |
|------|------|
| 情境 | 某访问凭据被泄漏（误提交到公开代码库、粘贴到聊天截图、设备遗失） |
| 触发 | 持有者一察觉立即发起吊销 |
| 期望 | 该凭据在秒级到分钟级失效；**其他凭据不受影响**；事后可查"哪把凭据最近被用过、用了多少次、是否出现异常地理位置"辅助风险评估 |
| 边界 | 若凭据被滥用期间已产生费用 / 订阅配额消耗，本需求不涵盖追偿；只保证止血 |

**典型画像**：
- 开发同事不小心 `git commit` 了含凭据的 `.env` 文件
- 助理的笔记本丢失
- 发现凭据被用在从未出现过的地区

#### S7：订阅凭据零下发

| 维度 | 内容 |
|------|------|
| 情境 | 持有者不希望原始订阅凭据（如第三方登录 token / OAuth refresh token）出现在任何终端设备上 |
| 触发 | 设计访问机制时 |
| 期望 | 终端只获得持有者签发的二级凭据；原始订阅凭据只存于受控服务端；即便终端被入侵也拿不到订阅本体 |
| 边界 | 服务端受控环境本身的安全性是另一个问题（参见 NFR-4） |

**典型画像**：
- 持有者提防受信用户从客户端反编译获取 token
- 学生使用凭据做作业，学生的机器毕业后不受控，期望毕业即失效但订阅本体无痕

### v2 场景

#### S8：多订阅池化

| 维度 | 内容 |
|------|------|
| 情境 | 单订阅容量不够（可能触发高频限流，可能面临团队扩张），持有者获得额外订阅（团队版多席位或多个个人订阅） |
| 触发 | 将多份订阅接入系统 |
| 期望 | 用户请求在多订阅间分发；某份订阅限流或失效时其他份承接；观察工具显示每份订阅的使用率、错误率、健康状态；持有者可禁用某份订阅观察影响 |
| 边界 | 对 MVP 阶段的单用户体验影响为零（仍然是"发一条、回一条"）；v2 中新会话 / 未绑定请求可切到其他订阅，已有 session 默认仍绑定首次命中的订阅，不做自动跨订阅迁移 |

**典型画像**：
- 3 人研究团队共用一份团队订阅，每人独立席位
- 持有者把自己的工作订阅 + 个人订阅都接入，池化承载家人使用

#### S9：高并发

| 维度 | 内容 |
|------|------|
| 情境 | 同时在线用户数超过单订阅容量，或个别凭据短时间内异常高频 |
| 触发 | 并发峰值 |
| 期望 | 系统公平调度；个别凭据超限时只限流自己，不拖垮其他用户；触发限流的用户看到清晰反馈（"当前请求已超频，N 分钟后再试"） |
| 边界 | 不做强业务优先级（所有用户平等）；高优先级保证是 v2+ 话题 |

**典型画像**：
- 教学班 15 人在同一堂课同时发问
- 会议演示 20 个听众同时点击

### v2+ 场景

#### S10：水平扩容 / 高可用

| 维度 | 内容 |
|------|------|
| 情境 | 系统规模 / 可靠性要求提升：30+ 并发或有 SLA 要求 |
| 触发 | 部署多个访问网关实例 + 前置负载均衡 |
| 期望 | 系统对用户透明扩容；单实例故障自动剔除；观察工具汇聚多实例数据 |
| 边界 | 需要引入共享状态（凭据、会话绑定、调度统计）；运维复杂度升级；进行中的流式响应允许短暂断连后重连 / 重试，不把"流式响应永不断线"作为默认承诺 |

**典型画像**：
- 从小圈子扩到 50+ 用户的机构内部服务
- 诊所 24×7 值班场景不能接受网关重启 5 分钟无服务

### 跨档次共性场景

#### S11：凭据例行轮换（常态安全卫生，不是应急）

| 维度 | 内容 |
|------|------|
| 情境 | 持有者按安全策略（如每 90 天）主动轮换凭据，不等到泄漏才换 |
| 触发 | 持有者签发新凭据 + 通知用户更换 + 旧凭据进入宽限期 + 宽限期后自动吊销 |
| 期望 | 轮换期间用户切换无感（凭据新旧并存一段时间）；忘记更换的用户看到清晰提示而不是神秘错误 |
| 边界 | S11 ≠ S6：例行 ≠ 应急；S11 要平滑，S6 要立即止血。例行轮换的新旧凭据应显式归属同一 subject，因此能共享同一用户的会话历史 |

**典型画像**：
- 持有者把 90 天轮换作为例行运维，避免凭据长期潜在泄漏

#### S12：同一用户跨设备会话延续

| 维度 | 内容 |
|------|------|
| 情境 | 同一用户（持有同一把凭据，或持有例行轮换后的同一 subject 新凭据）在设备 A 开始一段对话，稍后到设备 B 想看到这段对话并继续 |
| 触发 | 设备 B 输入同样的凭据，或输入同一 subject 下的新凭据 |
| 期望 | 设备 B 看到的会话列表 = 设备 A 看到的；点击可继续对话 |
| 边界 | MVP（单提供商、单订阅）支持最终一致；设备 B 可能需要刷新才能看到设备 A 的最新会话。v2 多订阅池化后同一 session 仍绑定首次命中的那份订阅，跨订阅合并不支持 |

**典型画像**：
- 医生在门诊开始的问诊，晚上回家接着看结果
- 学生在课堂做到一半的作业，晚上回寝室继续

#### S13：持有者观察与审计

| 维度 | 内容 |
|------|------|
| 情境 | 持有者定期或按需查看系统使用情况 |
| 触发 | 调用观察工具（命令、Web 页、仪表板，等等） |
| 期望 | MVP 至少可见每把凭据的活跃度、最近一次使用时间、错误率、当前限流状态、单订阅健康和系统总体请求率；v2 增加每份订阅的健康、错误、延迟 |
| 边界 | 不记录请求 / 响应的业务内容（隐私），只记录元数据 |

**典型画像**：
- 持有者每周检查一次，发现有凭据两个月没用 → 考虑回收
- 发现某凭据突然使用量飙升 → 判断是否滥用或被盗

## 5. 功能需求

### FR-1 访问凭据管理

1. 持有者可签发新访问凭据，一次性输出完整凭据供保存
2. 凭据属性：标签、类别（scope）、过期时间、频率限制
3. 持有者可列出所有已签发凭据（不暴露完整凭据值）
4. 持有者可吊销任何凭据，吊销秒级到分钟级生效
5. 凭据到达过期时间自动失效，无需手工处理
6. 持有者可为同一 subject 做例行轮换：签发新凭据、保留旧凭据宽限期、宽限期后自动失效

### FR-2 访问控制

1. 所有请求必须携带有效凭据才能使用 AI 能力
2. 无凭据 / 无效 / 吊销 / 过期 / 超限流 必须返回**结构化错误**，包含可机器识别的错误码与用户可读文本
3. 错误码集至少覆盖：缺凭据、凭据无效、凭据吊销、凭据过期、访问被拒绝（含限流）

### FR-3 授权与隔离

1. 不同类别（scope）的凭据对应不同权限边界
2. 权限边界在服务端强制，客户端无法绕过
3. 凭据之间的使用计量和速率限制相互独立
4. 会话历史归属 subject。不相关凭据默认对应不同 subject，彼此不可见；例行轮换产生的新旧凭据可归属同一 subject，并共享该 subject 的会话历史

### FR-4 订阅凭据托管

1. 订阅凭据（对 AI 服务的登录态）只存在服务端受控目录
2. 若提供商支持服务端续期，服务端应自动续期订阅凭据；若续期失败或提供商不支持自动续期，系统必须进入清晰的"需要重新授权"状态
3. 终端设备上不存在任何订阅凭据副本

### FR-4a 提供商中立

1. 访问网关核心对 AI 提供商无硬编码依赖；支持的典型提供商包括但不限于 OpenAI ChatGPT / Codex、Anthropic Claude Pro/Max、Kimi、DeepSeek
2. MVP 仅覆盖单个已验证提供商（见 D9）；接入新提供商必须通过 adapter 验证认证、续期、调用、流式响应、错误归一化和会话延续能力
3. 同一套访问凭据体系（签发 / 限流 / 分权 / 吊销）在不同提供商之间语义一致；模型能力、工具能力和会话能力的差异由 adapter 明确暴露，不能隐式假设一致
4. 不同订阅（无论同提供商多订阅、还是跨提供商混用）在观察工具里是独立条目；MVP 只有一份订阅，订阅维度观察以单条目呈现

### FR-5 客户端体验

1. 远程模式：客户端配置"服务地址 + 访问凭据"，两项信息即可接入
2. 本地模式：客户端支持脱离访问网关独立工作（详见 §5.5 "本地模式是否必要"）
3. 客户端在连接过程中能反馈清晰状态：加载中 / 就绪 / 凭据缺失 / 凭据失效 / 服务不可达
4. 切换凭据或地址无需重装客户端
5. 流式响应（逐字出现）体验不受网关中转影响
6. 同一 subject 的多设备会话列表至少最终一致；MVP 不要求近实时推送同步

### FR-6 观察与审计

1. 每次访问的元数据（不含完整凭据、不含敏感内容）可用于事后查阅
2. 每把凭据的活跃度、最近使用时间、错误率可查
3. MVP 可查服务端自身健康和单订阅状态；v2 可按每份订阅查看健康、错误和延迟

## 6. 非功能需求

### NFR-1 性能

- 网关自身转发开销需要独立可测；常规网络下端到端首包延迟增量目标为 < 300 ms，但验收必须注明网络条件
- 流式响应不被网关整包缓冲，客户端能持续看到增量输出
- 单用户长请求不阻塞其他用户

### NFR-2 可用性

- MVP 单实例故障最大恢复时间 < 10 分钟
- 凭据轮换期间不产生用户可感知中断
- MVP 网关升级允许短暂断连后客户端自动重连；v2+ 再要求多实例滚动升级期间已完成会话不丢失

### NFR-3 可扩展性

- MVP 承载凭据数 ≥ 10、活跃用户数 ≥ 5、提供商数 = 1、订阅数 = 1
- v2 承载凭据数 ≥ 200、活跃用户数 ≥ 50，并支持多订阅池化
- 从 MVP 扩到 v2 不要求破坏性迁移；允许幂等、自动、可回滚的小迁移

### NFR-4 安全

- 所有客户端—服务端流量强制 TLS
- 凭据值不明文落盘（落盘需加密或 hash）
- 日志、错误信息、指标中不出现完整凭据
- 高危管理端点有独立的访问控制，不向普通凭据开放

### NFR-5 运维

- 部署流程文档化到"生手 ≤ 4 小时照做即能跑通"
- 常见故障有明确事故预案
- 凭据轮换、订阅重新授权、单份订阅故障隔离、访问网关升级 均有可复用的运维流程

## 7. 约束

### C-1 合规由持有者自担

本系统不对第三方 AI 服务条款做法律承诺。持有者根据订阅类型、分发对象、风险偏好自行决定使用边界。

### C-2 非商业化

本系统不作为对外付费 SaaS 运营。签发凭据给非持有者可控的外部人员、或把访问能力打包出售，均超出本需求范围。

### C-3 医疗合规

涉及患者数据的医疗合规审计（HIPAA / 等保等）不在本需求范围。若后续需要，需另立合规需求文档。

## 9. 验收标准

### MVP（覆盖 S1–S7 + 跨档 S11–S13 基础版）

MVP 验收前提：单提供商、单订阅、多访问凭据；多订阅池化、多提供商并存、多网关实例不进入 MVP 验收。

- [ ] S1：3 台不同设备 ≤10 分钟完成配置；任一设备首条 AI 请求 ≤30 秒返回成功
- [ ] S2：陌生机器 ≤5 分钟完成安装和配置；签发的"1 小时过期"凭据在 65 分钟后访问被拒
- [ ] S3：3 把独立凭据同时在不同设备工作 1 小时无干扰；吊销任一把不影响其他
- [ ] S4：medical 凭据尝试高危操作应被拒（返回权限错误）；code 凭据应通过
- [ ] S5：签发低限流凭据（如每分钟 5 次），超限后第 6 次请求被拒，1 分钟后恢复
- [ ] S6：吊销一把凭据，10 秒内该凭据失效，其他凭据正常
- [ ] S7：协议、客户端配置、抓包、客户端持久化目录、日志与错误信息中均不出现订阅原始凭据；终端只持有访问凭据
- [ ] S11：同一 subject 下的新旧凭据并存 24 小时，旧凭据到达过期时间自动失效，用户切换过程中无可感知中断
- [ ] S12：同一 subject 在设备 A 开始的会话，在设备 B 用同凭据或同 subject 的轮换新凭据登录后可见、可继续
- [ ] S13：观察工具至少能报告每把凭据的 `last_used_at`、本周请求数、错误率
- [ ] NFR-1：在注明网络条件下，远程模式首包延迟相比本地模式增量 ≤ 300 ms（实测 5 次平均）；另有网关自身转发开销测试结果
- [ ] NFR-5：一位未参与开发的人按部署文档成功搭起整套系统 ≤ 4 小时

### v2（覆盖 S8–S9 + S11–S13 扩展版）

- [ ] S8：3 份订阅池化，10 个并发用户持续 1 小时，无整体故障
- [ ] S8：停用其中一份订阅，10 秒内新 session / 未绑定请求不再命中该订阅；已绑定该订阅的 session 返回清晰错误或等待恢复，不自动跨订阅迁移
- [ ] S9：峰值 30 个并发用户，单用户超频被限流不拖垮其他用户
- [ ] S12（v2 子情形）：多订阅池化下，同 session 始终路由到首次命中的那份订阅
- [ ] S13（v2 扩展）：观察工具能按订阅维度展示健康、错误、延迟
- [ ] NFR-3：v1 → v2 升级不要求破坏性迁移；如需要小迁移，必须幂等、自动、可回滚

### v2+（覆盖 S10）

- [ ] S10：3 个网关实例 + LB，任一实例故障后客户端可自动重连，已完成消息不丢失；指标汇聚可用
- [ ] NFR-2：滚动升级网关期间已完成会话不丢失；进行中的流式响应允许重连后重试或重新生成

## 10. 决策待定（影响产品范围）

以下是**产品层决策**，拍板后会改变 MVP 边界或验收标准。不是技术选型。

### D1：MVP 是否启用 scope 分级？

- **A**：MVP 默认单一 scope（code/全开），FR-3 退化为"所有凭据相同权限"，S4 推迟
- **B**：MVP 实现 medical / code 两档，S4 进入 MVP
- **默认假设**：B（MVP 涵盖 S4），scope 分级是核心价值

### D2：MVP 是否必须保留本地模式？

- **A**：保留（用户可选远程 / 本地）
- **B**：远程模式专用（没有本地回退）
- **默认假设**：A，本地模式零成本保留、作为降级路径

### D3：MVP 规模目标

- **A**：3–5 设备 / 2–3 用户（个人多设备为主）
- **B**：5–15 设备 / 3–10 用户（小圈子共享）
- **C**：更大
- **默认假设**：B（若选 C，NFR-3 承载量和 S8 的 v2 优先级都要调）

### D4：凭据吊销生效时间 SLA

- **A**：秒级（≤ 10 秒）
- **B**：分钟级（≤ 5 分钟）
- **默认假设**：A（对 S6 应急价值显著）

### D5：是否对外 "演示/临时授权" 作为 MVP 需求

- **A**：S5 进入 MVP 验收
- **B**：S5 延后到 v2
- **默认假设**：A（成本低，能力覆盖广）

### D6：凭据例行轮换（S11）宽限期多长

- **A**：24 小时（平衡安全与不中断）
- **B**：72 小时（给忘记更换的用户更多缓冲）
- **C**：1 周（长缓冲，但潜在泄漏暴露窗口也长）
- **默认假设**：A

### D7：观察工具（S13）粒度

- **A**：只按凭据维度（谁、最近何时、多少次）
- **B**：凭据维度 + 订阅维度（S13 扩展版必须，v2 阶段）
- **C**：凭据维度 + 订阅维度 + 每 session 级（最细，隐私敏感）
- **默认假设**：MVP 选 A，v2 升到 B，C 仅在有明确故障排查需求时开启

### D8：S12 跨设备会话延续的一致性预期

- **A**：最终一致（设备 B 可能需要手动刷新才看到设备 A 最新对话；同一 subject 下的轮换新凭据可继承会话历史）
- **B**：近实时（设备 B 秒级内收到设备 A 更新）
- **默认假设**：A（MVP）；B 作为 v2 增强

### D9：MVP 提供商覆盖范围

- **A**：MVP 先覆盖**一个已验证**提供商（最熟悉的一家，如 OpenAI ChatGPT / Codex），打通全链路再扩
- **B**：MVP 同时覆盖 2–3 个主流提供商（OpenAI + Claude 之类），证明中立抽象
- **C**：MVP 覆盖所有计划支持的（OpenAI / Claude / Kimi / DeepSeek 全上），周期最长
- **默认假设**：A。理由：先用一个走通，并验证 adapter 合同；其他提供商按同一套抽象追加。B/C 把不必要的复杂度前置到 MVP，会拖慢 go/no-go 验证
- **影响**：A 的话 v2 要加一条"提供商扩展"工作（把第二、第三个提供商接入同一套抽象）

## 11. 术语

| 术语 | 含义 |
|------|------|
| **提供商** | AI 服务的供应方。典型包括 OpenAI（ChatGPT / Codex）、Anthropic（Claude Pro / Max）、Moonshot（Kimi）、DeepSeek 等；本文刻意不偏好任何一家 |
| **订阅** | 持有者在某个提供商购买的付费服务（含按账号登录使用的权利）。一个持有者可以有多份订阅，跨不同提供商 |
| **订阅凭据** | 对提供商服务的登录态（OAuth token / subscription bearer / API key 等，视提供商而定） |
| **访问凭据** | 持有者签发给终端的二级凭据（本系统自己签发，与提供商无关）。访问凭据归属某个 subject |
| **subject** | 访问网关内的逻辑使用者 / 会话归属单位。默认一把新凭据对应一个新 subject；例行轮换时新旧凭据可归属同一 subject |
| **持有者** | 订阅所有者（§3） |
| **终端用户** | 持访问凭据使用 AI 能力的人（§3） |
| **访问网关** | 本需求涉及的受控服务端组件（统称） |
| **scope** | 访问凭据的权限类别（如 medical、code） |
| **会话** | 用户与 AI 的一次连续交互上下文。会话归属 subject，并在 v2 多订阅池化时绑定首次命中的订阅 |
| **限流** | 基于时间窗口的使用频率限制 |
| **吊销** | 使一把访问凭据立即失效 |
